Chapter 02

AI Agents

What makes something an "agent"?

What Makes an AI an Agent?

Four things that separate an agent from a chatbot:

01 / TOOLS

Can call APIs, run code, read files

02 / LOOP

Thinks → Acts → Observes → Repeats

03 / GOAL

Pursues an objective, doesn't just respond

04 / MEMORY

Retains context across steps

A chatbot answers once and stops.

An agent keeps going.

Four things define an agent:

- Tools → it can act (API, DB, code)
- Loop → it keeps trying
- Goal → it is trying to complete something
- Memory → it remembers state

Example:

Fix a bug.

Chatbot:
→ explains error

Agent:
→ fixes code
→ runs tests
→ retries if failing

Key point:
The loop is everything.

No loop → not an agent.

Chapter 03

Agentic AI

When AI pursues goals, not just tasks

Agent vs. Agentic

AI AGENT

  • Given a specific task
  • Has a defined end point
  • "Search these docs and summarise"

AGENTIC AI

  • Given a broad goal
  • Plans its own steps
  • "Launch this feature by Friday"

Analogy: An agent is like a contractor who builds what you spec. Agentic AI is like a project manager — it hires, specs, and delivers.

This is where most people get confused.

Agent:
- given steps
- executes task
- stops

Agentic AI:
- given goal
- decides steps
- adapts when things fail

Example:

Agent:
"Summarise this file"

Agentic:
"Ship this feature"

It will:
- explore code
- write code
- test
- fix
- retry

Key point:
Agent follows instructions.
Agentic figures things out.

Multi-Agent Systems

When agents work together

Why Multiple Agents?

One agent hits limits. A team breaks through them:

CONTEXT LIMITS

One agent can't hold an entire large codebase at once

PARALLELISM

Run ten tasks at the same time instead of one by one

SPECIALISATION

A security agent, a code agent, a docs agent — each an expert

CROSS-CHECKING

Agent A answers, Agent B critiques — fewer mistakes

The pattern: one agent plans → others execute in parallel. Like a manager and a team.

One agent hits limits.

- context limits
- slow execution
- no specialization

Multi-agent solves this.

Example: build a feature

Chapter 06

Failures & Risks

What actually goes wrong

Things That Actually Went Wrong

Agent deleted an entire database table

Asked to "get rid of" a record — deleted the whole table. Ambiguous language plus irreversible action.

Infinite loop — $2,000 API bill overnight

Agent kept retrying a stuck task with no limit set. The bill arrived before anyone noticed.

MCP tool poisoning — data was stolen

A malicious MCP server tricked an agent into sending private data to an external endpoint.

Errors compound across agent pipelines

Agent A is wrong → Agent B builds on it → Agent C reports the wrong answer confidently.

Agents fail in real ways.

Not theory.

Examples:

- Deleted entire database
  → vague instruction

- Infinite loop
  → huge API bill

- Data leak
  → malicious tool

- Error chaining
  → wrong → worse → confident wrong

Key point:
Agents amplify mistakes.

How to Build Safer Agents

HUMAN IN THE LOOP

Require approval before any irreversible action

LEAST PRIVILEGE

Only grant the tools the agent actually needs

CIRCUIT BREAKERS

Set max retries and spending limits before you deploy

AUDIT LOGS

Always know exactly what the agent did and why

You need guardrails.

- Human approval for critical actions
- Limit tool access
- Set retry and cost limits
- Log everything

Example:
Delete operation → always require approval

Key point:
If you cannot control it, do not deploy it.
