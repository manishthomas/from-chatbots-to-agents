Chapter 03

Agentic AI

When AI pursues goals, not just tasks

[[ chapter opener ]]

Agent vs. Agentic

AI AGENT

  • Given a specific task
  • Has a defined end point
  • "Search these docs and summarise"

AGENTIC AI

  • Given a broad goal
  • Plans its own steps
  • "Launch this feature by Friday"

Analogy: An agent is like a contractor who builds what you spec. Agentic AI is like a project manager — it hires, specs, and delivers.

An agent does what you tell it, step by step. Agentic AI breaks the goal into steps itself — that is the difference. [[ read the analogy out loud — contractor vs project manager ]] One builds what you spec. The other specs it, hires, and delivers.

Chapter 07

Failures & Risks

What actually goes wrong

[[ chapter opener — shift tone here: honest, not scary ]]

Things That Actually Went Wrong

Agent deleted an entire database table

Asked to "get rid of" a record — deleted the whole table. Ambiguous language plus irreversible action.

Infinite loop — $2,000 API bill overnight

Agent kept retrying a stuck task with no limit set. The bill arrived before anyone noticed.

MCP tool poisoning — data was stolen

A malicious MCP server tricked an agent into sending private data to an external endpoint.

Errors compound across agent pipelines

Agent A is wrong → Agent B builds on it → Agent C reports the wrong answer confidently.

[[ click through each failure card — let each one land before moving on ]] These are real incidents, not hypotheticals. The database deletion: ambiguous language plus irreversible action. The loop: no cost limit set. Bill arrived before anyone noticed. MCP poisoning: a new attack surface most teams are not thinking about yet. Error compounding: the scariest one — confident wrong answers, all the way down. [[ the lesson: safety is not optional, it is engineering ]]

How to Build Safer Agents

HUMAN IN THE LOOP

Require approval before any irreversible action

LEAST PRIVILEGE

Only grant the tools the agent actually needs

CIRCUIT BREAKERS

Set max retries and spending limits before you deploy

AUDIT LOGS

Always know exactly what the agent did and why

[[ click through each card ]] Start with human-in-the-loop. Remove it only when you trust the agent in that specific context. Least-privilege: only give the agent what it absolutely needs. Circuit breakers: set a max spend and max retries before you deploy — not after. Audit logs: if you cannot explain what the agent did, you cannot fix it when it goes wrong. [[ these are engineering patterns, not theory — you can implement all four this week ]]

You Already Have the Right Mental Model

As React developers, you already think in systems — not just lines of code:

REACT THINKING

  • Components with clear inputs & outputs
  • State flows in one direction
  • Side effects are isolated
  • Compose small pieces into large systems

AGENT THINKING

  • Agents with clear goals & tool access
  • Information flows between agents
  • Actions are bounded & observable
  • Compose specialists into pipelines

Your instinct for component boundaries, data flow, and side-effect isolation — that is exactly the mental model for building reliable agent systems.
